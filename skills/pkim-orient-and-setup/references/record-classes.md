# Record classes

PKIM defines four record classes. Every record you touch is one of these.

## EV — Evidence

Original source material: PDFs, web captures, imported documents, scanned pages, formatted notes, whatever the operator captured. Evidence records are the atoms; every knowledge claim traces back to one or more EVs.

- **PKIM_ID format:** `EV-YYYYMMDD-NNNN` (class-date-sequence)
- **Databases:** `PKIM-Evidence-Personal`, `PKIM-Evidence-Work`, `PKIM-Evidence-Server`, `PKIM-Pilot`. Never in `PKIM-Knowledge`.
- **DT record type:** whatever the source was — usually `pdf`, `html`, `webarchive`, `markdown`, `formatted_note`.
- **DocRole:** `evidence`
- **Filing:**
  - `/Inbox` on capture
  - `/Sources/Imported` for imported records (file lives inside `.dtBase2`)
  - `/Sources/Indexed` for indexed records (file lives on disk, DT references it)
  - `/Captures/*` for web / bookmark / scan captures
  - `/Working` while being reviewed
  - `/Review` when ready for approval
  - `/Archive` after retirement

## KN — Knowledge Note

Native markdown notes that describe what evidence *means*. The synthesis surface. A KN is authored by a human (or agent with review) after reading one or more EVs; it does not exist in the wild.

- **PKIM_ID format:** `KN-YYYYMMDD-NNNN`
- **Database:** `PKIM-Knowledge` only.
- **DT record type:** `markdown` (indexed against the on-disk knowledge root; the file is the canonical body).
- **DocRole:** `knowledge`
- **NoteType** (custom metadata, one of): `literature` / `synthesis` / `topic` / `project` / `decision` / `workflow`
- **Filing:**
  - `/Notes/Literature` — literature notes (one KN per EV, close reading)
  - `/Notes/Synthesis` — synthesis notes (many EVs → one argument)
  - `/Notes/Topics` — topic notes (define what a concept means and its boundary)
  - `/Notes/Projects` — project notes (goal / context / status)
- **Body shape:** MultiMarkdown frontmatter (Title / PKIM_ID / DocRole / NoteType / Review_State / Aliases / PrimaryTopic) then `# Title`, `## Summary`, `## Claims`, `## Evidence links`, `## Related notes`.

## RL — Relation Note

First-class edge records. When one record supports / contradicts / extends / summarises / references / exemplifies / precedes / supersedes another, that edge is an RL record — not a WikiLink in prose, not a bullet in a list. RLs are the graph structure.

- **PKIM_ID format:** `RL-YYYYMMDD-NNNN`
- **Database:** `PKIM-Knowledge` only (edges live with the knowledge, not with the evidence).
- **DT record type:** `markdown`
- **DocRole:** `relation`
- **Relation_Type** (custom metadata, closed set): `supports` / `contradicts` / `extends` / `summarizes` / `references` / `exemplifies` / `precedes` / `supersedes`
- **Filing:** `/Notes/Relations`
- **Endpoints:**
  - `Source_Item` (custom metadata): item link (`x-devonthink-item://<uuid>`) to the source record. Cross-database references (from KN to EV) are the reason item links exist.
  - `Target_Item`: same shape.
- **Body shape:** frontmatter (Title / PKIM_ID / DocRole / Relation_Type / Source_Item / Target_Item / Review_State / RelationStatus) then `# Why this relation exists` (mandatory prose rationale — an RL without rationale is invalid) then optional `## Interpretation`.

## CL — Claim

Individual claims promoted out of a KN's `## Claims` block into their own records. This is the "claims as nodes" model — a KN's Claims section becomes a bullet list of WikiLinks to CL records rather than an inline YAML block.

- **PKIM_ID format:** `CL-YYYYMMDD-NNNN`
- **Database:** `PKIM-Knowledge` only.
- **DT record type:** `markdown`
- **DocRole:** `claim`
- **ClaimType** (custom metadata, closed set): `fact` / `inference` / `assumption` / `open-question`
- **ClaimConfidence** (custom metadata, closed set): `low` / `medium` / `high`
- **Filing:** `/Notes/Claims`, optionally grouped by parent KN (`/Notes/Claims/<parent-KN-name>`).
- **Parent:** the CL body has a `## Parent` section with a WikiLink to its parent KN.
- **Body shape:** frontmatter (Title / PKIM_ID / DocRole / ClaimType / ClaimConfidence / Review_State / ParentKN_ID) then `# The claim` (one-sentence statement), `## Reasoning` (why this claim holds, with cited EVs), `## Evidence` (WikiLinks or item links to EV records), `## Parent` (WikiLink to the KN this claim was extracted from), `## Contradicted by` (optional).

## PKIM_ID minting

`<CLASS>-<YYYYMMDD>-<NNNN>` where NNNN is the next unused sequence for that class + date pair.

To mint one:

1. Determine class (EV / KN / RL / CL).
2. Determine date (usually today's UTC date, `YYYYMMDD`).
3. Query DT for existing IDs of that class+date: `mcp__devonthink__search_records` with `query: "mdpkim_id:<CLASS>-<DATE>-*"`.
4. Parse the highest sequence number seen; add 1. If none found, start at `0001`.
5. Format: zero-padded 4 digits.

The DT UUID is separately generated by DEVONthink on record creation. It is the persistent identifier for cross-references. `PKIM_ID` is the human-readable index.

## Alias convention

Every record's DT `aliases` field carries its `PKIM_ID` (semicolon-joined with the display name, e.g. `Purpose Design; KN-20260429-0002`). This makes `PKIM_ID` searchable via `mcp__devonthink__lookup_records name:`, and it means an operator glancing at DT sees the ID next to the name.

## When to use each class

- **Captured something new** → EV
- **Read an EV, forming my own understanding** → KN
- **Two records are related in a semantic way that matters** → RL
- **A specific claim inside a KN needs to be individually addressable** (search it, contradict it, audit it) → CL

Do not: create a KN that just paraphrases one EV. Do not: create an RL for every WikiLink in prose (only semantically load-bearing edges). Do not: extract every bullet in a KN's `## Claims` block into a CL — only the claims that need to be individually queryable.
